| Layer | Description | Example |
|-------|-------------|---------|

### Four Major CLI Coding Agents Evaluation

| Agent | Score | Context | Sandbox | License | Best For |
|-------|-------|---------|---------|---------|----------|
| **Claude Code** | ★★★★★ | 1M tokens | Seatbelt/Bubblewrap | Proprietary $20-100/mo | Complex refactoring, deep governance |
| **OpenCode** | ★★★★☆ | varies by provider | Docker | MIT (97.5K ⭐) | Multi-providers, free, flexible |
| **Codex CLI** | ★★☆☆☆ | 272K-1.05M | Seatbelt/Landlock | Apache 2.0 | Basic agent, PR review |
| **Gemini CLI** | ★★☆☆☆ | 1M tokens | Docker | Apache 2.0 (free 1000 req/day) | Free trial, reliability issues |

**Source**: Patrick Hulce tested 10 coding agents on same SaaS project (Hulkastorus); Claude Code best but still has context saturation causing repeated mistakes.

**Architecture difference** (Blake Crosley analysis):
- Codex: Kernel-level sandbox (OS syscall level) — strong boundary, coarse granularity
- Claude Code: Application-layer hooks (26 programmable events) — medium boundary, fine granularity

### Hubert's Decisions

1. **Budget**: Continue with OpenCode (multi-providers); evaluate Claude Code when large deep development needs arise
2. **Multi-agent integration**: External coding agent integration not high ROI — tools are closed, ecosystems fragmented. OpenCode's multi-providers already does分流. External agent = "supplement" not "replacement"
3. **External coding agent positioning**: OpenClaw = personal assistant, coding agent = software dev specialist. Different positioning, need external agent for software development professional tasks

### Spawn Subagent vs External Coding Agent

| Dimension | Spawn Subagent | External Agent |
|-----------|---------------|---------------|
| Permissions | Shared parent tool permissions | Independent sandbox (Codex: syscall level) |
| Context | Shared parent capacity | Independent large context window |
| Info return | Direct write to parent session | Need `openclaw message send` notification |
| Integration complexity | Low | High (need inject notification route) |
| Tool calls | All OpenClaw tools | Can only use own toolset |

### Workspace Renovation Direction

**Current architecture is already Lazy Load**:
- AGENTS.md (3.4KB) only does directory index; Skills/Workers not auto-loaded at startup
- Bootstrap inject has truncation (AGENTS.md only ~25% of injected context)

**Suggested new structure**:
```
workspace/
├── orchestration/          ← NEW: explicit orchestration policy (lazy load)
│   ├── CODING.md          ← External coding agent usage guide
│   ├── SUBAGENTS.md       ← Subagent spawn detailed protocol (extracted from WORKERS.md)
│   └── INTEGRATION.md     ← OpenClaw ↔ External agent info exchange protocol
├── WORKERS.md             ← Can be condensed, only keep discovery logic
├── RULES.md               ← Coding Mode section can be moved to orchestration/
└── AGENTS.md              ← Keep as-is (already lean)
```
